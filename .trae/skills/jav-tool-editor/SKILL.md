# Jav工具库脚本编辑器技能

## 技能目的

专门用于维护和修改"Jav工具库"油猴脚本，确保所有修改都遵循既定的开发规范和工作流程。

## 何时使用此技能

- 当用户要求修改Jav工具库脚本时
- 当用户要求添加新功能到Jav工具库时
- 当用户要求修复Jav工具库的bug时
- 当用户要求优化Jav工具库的性能或样式时

## 核心原则

### 1. 版本号管理规则

版本号格式为 **X.X.X**（三位数字）：

- **个别bug调试**（没有加入新功能）：**不需要改动版本号**
- **小型功能优化**：**递增第三个X**（例如：7.0.2 → 7.0.3）
- **多个功能优化、次层级功能增加、样式调整**（如修改配置菜单中的某一子功能）：**递增第二个X**（例如：7.0.2 → 7.1.0）
- **大版本更新**（加入重量级功能）：**递增第一个X**（例如：7.0.2 → 8.0.0）

### 2. 单行化规则

- **所有新增代码**，在确保脚本可用的情况下，**严格执行单行化**
- 单行化目的是压缩纵向空间，不改变现有功能
- 单行不能过长，否则可能会导致油猴插件无法识别
- 参考脚本现有单行化的代码风格

### 3. 样式协调统一

- 前端样式类修改，请**遵循现有的CSS设置**
- 尽量保持新功能和脚本现有样式协调统一
- 涉及样式（按键布局、颜色）时，**需给用户提供样式示例**

### 4. 最小改动原则

- 修复问题时，采用最小改动模式
- 避免不必要的代码变更
- 保持代码的稳定性

### 5. 先方案后修改

- 每次用户提出需求，**需先给用户方案**
- 说明需要用户补充的信息
- **待用户同意后再修改**

## Git同步工作流程

### 初始设置

脚本已配置Git仓库：
- 本地仓库路径：`/Users/hefang/Documents/trae_projects/Jav-tool`
- 远程仓库：`https://github.com/Harpenter/Jav-tool.git`

### 修改后推送流程

每次修改完成后：
1. 检查工作区状态：`git status`
2. 添加修改：`git add .`
3. 提交修改：`git commit -m "有意义的提交信息"`
4. 推送到远程：`git push origin main`

### 保留历史版本

- 所有版本都会保存在GitHub仓库的提交历史中
- 用户可以随时查看或回滚到任意版本
- 不要删除历史提交

## 脚本元数据更新

修改脚本时，注意更新以下元数据：

```javascript
// ==UserScript==
// @name         Jav工具库
// @version      X.X.X
// @description  ...
// @homepageURL  https://github.com/Harpenter/Jav-tool
// @updateURL    https://raw.githubusercontent.com/Harpenter/Jav-tool/main/Jav工具库-X.X.X.user.js
// @downloadURL  https://raw.githubusercontent.com/Harpenter/Jav-tool/main/Jav工具库-X.X.X.user.js
// ==/UserScript==
```

确保：
- 文件名与版本号一致（例如：`Jav工具库-7.0.2.user.js`）
- `@updateURL` 和 `@downloadURL` 指向正确的Raw文件

## 工作流程

### 步骤1：理解需求

当用户提出需求时：
1. 仔细阅读用户需求
2. 检查脚本当前状态
3. 确定需要修改的范围

### 步骤2：给出方案

在开始修改之前：
1. 给出具体的修改方案
2. 说明需要用户补充的信息
3. 涉及样式时提供样式示例
4. 等待用户同意

### 步骤3：实施修改

用户同意后：
1. 使用TodoWrite规划任务
2. 按照方案进行修改
3. 确保新增代码单行化
4. 保持样式协调

### 步骤4：Git提交

修改完成后：
1. 检查版本号是否需要更新
2. 如需要，更新文件名和URL
3. 提交到本地Git仓库
4. 推送到GitHub远程仓库

### 步骤5：验证结果

推送成功后：
1. 确认GitHub仓库已更新
2. 提供给用户新的Raw URL（如需要）
3. 告知用户后续使用方法

## 注意事项

1. **不要破坏现有功能**：修改时确保不影响已有功能的正常运行
2. **测试修改**：修改后尽可能测试新功能是否正常工作
3. **遵循现有代码风格**：保持代码风格一致
4. **及时备份**：Git提交本身就是备份，确保每次有意义的修改都提交
5. **保持版本一致性**：文件名、脚本内版本号、URL中的版本号必须一致
