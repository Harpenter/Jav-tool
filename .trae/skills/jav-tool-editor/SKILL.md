# Jav工具库脚本编辑器技能

## 技能目的

专门用于维护和修改"Jav工具库"油猴脚本，确保所有修改都遵循既定的开发规范和工作流程。

## 何时使用此技能

- 当用户要求修改Jav工具库脚本时
- 当用户要求添加新功能到Jav工具库时
- 当用户要求修复Jav工具库的bug时
- 当用户要求优化Jav工具库的性能或样式时

## 核心原则

### 1. 版本号管理规则

版本号格式为 **X.X.X**（三位数字）：

- **个别bug调试**（没有加入新功能）：**不需要改动版本号**
- **小型功能优化**：**递增第三个X**（例如：7.0.2 → 7.0.3）
- **多个功能优化、次层级功能增加、样式调整**（如修改配置菜单中的某一子功能）：**递增第二个X**（例如：7.0.2 → 7.1.0）
- **大版本更新**（加入重量级功能）：**递增第一个X**（例如：7.0.2 → 8.0.0）

### 2. 单行化规则

- **所有新增代码**，在确保脚本可用的情况下，**严格执行单行化**
- 单行化目的是压缩纵向空间，不改变现有功能
- 单行不能过长，否则可能会导致油猴插件无法识别
- 参考脚本现有单行化的代码风格
- 回调函数、Promise 链式调用也需单行化
- 条件分支中的单语句块可合并为单行

### 3. 样式协调统一

- 前端样式类修改，请**遵循现有的CSS设置**
- 尽量保持新功能和脚本现有样式协调统一
- 涉及样式（按键布局、颜色）时，**需给用户提供样式示例**

### 4. 轻量化实现原则（重要）

这是从多轮对话中提取的核心原则：

- **方案先行原则**：任何修改前必须先给出清晰的技术方案，包括：
  - 问题根源分析
  - 至少 2 个可行方案对比
  - 推荐方案及理由
  - 待用户明确同意后再执行修改
  
- **轻量化实现原则**：
  - 优先使用浏览器原生 API（如 sessionStorage、Map）
  - 避免新增全局变量或 GM 存储
  - 最小化修改现有代码结构
  - 如需新增功能，先评估是否能复用现有逻辑
  - 例如：需要临时存储时优先用 sessionStorage 而非 GM_setValue

### 5. 缓存策略规范（重要）

这是从下载队列磁链混淆问题中提取的关键规则：

- **缓存隔离规则**：
  - 使用 Map 结构存储 per-item 缓存（key 为唯一标识如 avId）
  - 禁止使用全局对象存储缓存（避免多页面/多标签污染）
  - 明确数据源：网络请求 vs 页面 DOM，两者不可混用
  
- **数据源使用规则**：
  - 下载队列等批量操作：仅使用网络请求数据
  - 详情页等单页面操作：可使用页面 DOM + 网络请求
  - 跨页面场景：必须隔离数据源，防止相互干扰
  - 如果函数接收可选的 document 参数，必须明确判断是否传入，而非使用 `|| document` 回退

### 6. 先方案后修改

- 每次用户提出需求，**需先给用户方案**
- 说明需要用户补充的信息
- **待用户同意后再修改**

## 问题诊断流程（重要）

从问题定位中提取的系统性方法：

- **问题定位步骤**：
  1. 复现问题场景（用户描述的具体操作流程）
  2. 定位相关代码段（使用 SearchCodebase 或 Grep）
  3. 分析数据流向（缓存如何写入/读取）
  4. 识别污染源（哪个环节导致数据混淆）
  5. 给出根因分析（用通俗语言解释给用户）

- **验证修复效果**：
  - 说明修复后的预期行为
  - 提示用户如何测试验证
  - 必要时添加调试日志或验证步骤

- **典型问题模式**：
  - 多标签页数据混淆 → 检查是否使用了全局变量而非 Map
  - 缓存数据错误 → 检查数据源是否正确隔离
  - 跨页面干扰 → 检查是否有函数使用了 `|| document` 回退

## Git同步工作流程

### 初始设置

脚本已配置Git仓库：
- 本地仓库路径：`/Users/hefang/Documents/trae_projects/Jav-tool`
- 远程仓库：`https://github.com/Harpenter/Jav-tool.git`

### 修改后推送流程

每次修改完成后：
1. 检查工作区状态：`git status`
2. 添加修改：`git add .`
3. 提交修改：`git commit -m "vX.X.X: 简要描述"`
4. 推送到远程：`git push origin main`
5. **验证推送成功**：
   - 检查 git push 输出确认无错误
   - 可选：访问 GitHub 仓库页面或 raw URL 确认更新

### 保留历史版本

- 修改前备份当前版本：复制主脚本为 `Jav工具库-X.X.X.user.js`
- 所有版本都会保存在GitHub仓库的提交历史中
- 用户可以随时查看或回滚到任意版本
- 不要删除历史提交

### 提交信息规范

- 格式：`vX.X.X: 简短描述`
- 必须包含版本号
- 描述要清晰说明修复的问题或新增的功能

## 脚本元数据更新

修改脚本时，注意更新以下元数据：

```javascript
// ==UserScript==
// @name         Jav工具库
// @namespace    http://tampermonkey.net/
// @version      X.X.X
// @description  ... vX.X.X: 简要更新内容
// @author       Harpenter
// @homepageURL  https://github.com/Harpenter/Jav-tool
// @updateURL    https://raw.githubusercontent.com/Harpenter/Jav-tool/main/Jav工具库.user.js
// @downloadURL  https://raw.githubusercontent.com/Harpenter/Jav-tool/main/Jav工具库.user.js
// ==/UserScript==
```

**重要**：为确保 Tampermonkey 能检测到更新，`@updateURL` 和 `@downloadURL` 应使用固定文件名（不带版本号），例如 `Jav工具库.user.js`。

## 工作流程

### 步骤1：理解需求

当用户提出需求时：
1. 仔细聆听用户描述的问题或需求
2. 如有不明确之处，主动询问确认
3. 记录关键信息（问题场景、期望行为、当前行为）

### 步骤2：问题分析

1. 使用 SearchCodebase 或 Grep 定位相关代码
2. 分析数据流向和函数调用链
3. 识别问题根源（不要只修复表面症状）
4. 评估影响范围（是否影响其他功能）

### 步骤3：给出方案

向用户清晰阐述：
1. **问题根源**：用通俗语言解释为什么会出问题
2. **可行方案**：列出至少 2 个解决方案
3. **推荐方案**：说明选择该方案的理由
4. **预期效果**：修改后的预期行为
5. **验证方式**：用户如何测试确认问题已修复

**重要**：必须先得到用户同意后再进行修改，不要直接开始编码。

### 步骤4：实施修改

得到用户同意后：
1. 创建备份文件（如 `Jav工具库-7.0.5.user.js`）
2. 按最小改动原则修改代码
3. 确保单行化格式正确
4. 更新版本号和描述

### 步骤5：验证和同步

1. 检查代码语法正确性
2. 提交 Git 并推送到远程
3. 验证推送成功
4. 向用户总结完成的工作

## 错误处理规范

- 异步操作必须包含 try-catch
- 错误提示要友好（用户可见的文本）
- 关键操作失败要有回退机制
- 使用合理的变量命名，避免作用域混乱

## 沟通方式

- 技术问题用通俗语言解释（避免过多术语）
- 复杂问题用示例说明（如"比如你在下载队列中添加了 10 个文件..."）
- 修改完成后用清单体总结（清晰列出完成的工作）
- 涉及用户确认的节点必须明确等待用户同意
